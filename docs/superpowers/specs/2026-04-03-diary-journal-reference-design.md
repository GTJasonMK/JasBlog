# Diary Journal Reference Design

## 背景

当前 `JasBlog` 的考研日志详情页虽然已经支持按天聚合和多条 entry 展示，但整体视觉更接近通用内容卡片列表，而不是“当天日志页”。主要问题有两类：

1. 页面结构偏资讯详情页，缺少日志应有的连续阅读感、时间推进感和手写记录气质。
2. 日志内容无法自然联动已有的学习笔记和知识图谱。即使作者在正文里提到某篇 note 或 graph，也只能靠普通链接跳转，打断日志阅读。

本设计要解决的不是“再做一个复杂内容系统”，而是把 `diary` 调整为更像日志的阅读体验，并让日志里的引用能够在当前页面内完成辅助阅读。

## 目标

- 将 `diary` 详情页重构为更像“纸页日志”的阅读界面。
- 保持日志正文为连续文字阅读，不在正文中内嵌实时渲染的引用内容。
- 支持在 diary 正文中引用 `note` 与 `graph`，点击后通过站内浮层预览内容。
- 不引入新的 Markdown 语法，不改变现有内容契约。
- 不把本次能力扩散到全站其他页面。

## 非目标

- 不修改 `JasBlogEditor` 的内容写法和 frontmatter 契约。
- 不新增 `[[note:slug]]`、`[[graph:slug]]` 等自定义语法。
- 不将内部内容预览弹窗扩展到 note、project、roadmap、graph 等其他详情页。
- 不把 diary 页面做成“仪表盘”或“数据看板”风格。
- 不在日志正文中常驻展示 note 或 graph 的复杂嵌入块。

## 内容契约

日志中引用继续使用标准 Markdown 内链：

- `note`：`[状态同步排查笔记](/notes/react-state-sync)`
- `graph`：`[状态流转图谱](/graphs/state-flow)`

这样做的理由：

- 内容仍然是标准 Markdown，可读、可迁移、可直接在仓库中理解。
- 避免编辑器与站点出现一套新的引用语法分叉。
- 即使未来关闭弹窗能力，这些链接依然是合法的站内链接。

只有 `diary` 详情页中的 `/notes/<slug>` 与 `/graphs/<slug>` 两类链接会被识别为“可预览引用”。其他链接继续走当前 Markdown 链接行为。

## 视觉设计

### 页面整体

`diary` 详情页采用“日志纸页”方向，而非通用卡片方向：

- 顶部区域像“当天封面”，突出日期、记录数、情绪/天气/地点等日志上下文。
- 主体正文区域像铺开的纸页，带更强的连续阅读感。
- 多条 entry 仍按时间顺序展开，但每条 entry 更像同一页里的时间段落，而不是彼此独立的卡片。

### 单条 entry

每条 entry 的视觉层级如下：

- 时间
- 标题
- 正文段落
- 可选标签/上下文信息

重点是让 entry 成为日志叙事的一部分，而不是功能模块。正文应该更像“我今天做了什么、怎么想的、遇到了什么”的自然记录。

### 引用文字

引用在正文里只作为“被标识的文字”出现：

- 样式比普通链接更明显，但仍嵌在行文中
- 可带轻量的浅底色、细边框、箭头或类型提示
- 不在正文流中直接展开 note 或 graph 预览

这保证 diary 保持纯文字阅读体验，引用只是辅助入口，不破坏阅读节奏。

## 交互设计

### 打开预览

当用户点击 diary 正文中的引用文字：

1. 若链接是 `/notes/<slug>` 或 `/graphs/<slug>`，阻止默认跳转。
2. 在当前 diary 页面上方打开站内浮层。
3. 浮层根据引用类型展示对应的只读预览内容。

### 关闭方式

浮层必须支持以下关闭路径：

- 点击右上角关闭按钮
- 点击遮罩层
- 按 `Esc`

关闭后返回原 diary 页面，不重置当前阅读位置。

### 预览内容边界

`note` 浮层显示：

- 标题
- 日期
- 标签
- 正文内容

不显示：

- 返回列表链接
- 评论区

`graph` 浮层显示：

- 标题
- 描述
- 图谱画布
- 可选补充正文

不显示：

- 返回列表链接
- 独立详情页顶部提示栏

### 错误处理

如果引用目标不存在、slug 无效或类型不支持：

- 保留原始引用文字
- 点击后打开明确的错误浮层
- 错误文案需要直接说明未找到对应 note 或 graph

不允许静默降级为“什么都不发生”或偷偷跳走。

## 架构边界

本次实现限定在 `JasBlog` 内部，推荐拆分为以下职责：

- `DiaryEntryArticle`
  - 单条 entry 的展示组件
  - 负责时间、标题、纯文字正文和引用文字样式

- `DiaryReferenceMarkdown`
  - diary 场景专用 Markdown 渲染器
  - 只拦截 `/notes/*`、`/graphs/*` 两类内部链接
  - 其他 Markdown 行为尽量复用现有站点规则

- `DiaryReferenceModal`
  - 浮层容器
  - 负责遮罩、滚动、关闭逻辑、键盘交互

- `DiaryReferencePreview`
  - 根据 `{ type, slug }` 渲染引用预览
  - 内部再按 note / graph 分支展示

这样做可以把 diary 专属能力局限在 diary 页面，不污染现有全局 `MarkdownRenderer` 的默认行为。

## 数据流

1. `diary/[slug]` 页面读取聚合后的 day 数据。
2. 页面把每条 entry 交给 `DiaryEntryArticle`。
3. `DiaryEntryArticle` 使用 `DiaryReferenceMarkdown` 渲染正文。
4. 用户点击引用文字后，组件写入本地状态 `{ type, slug }`。
5. `DiaryReferenceModal` 打开。
6. `DiaryReferencePreview` 根据类型加载并展示 note / graph 数据。
7. 用户关闭浮层后，状态清空，回到 diary 阅读流。

## 验证要求

至少需要覆盖以下验证：

- diary 正文中的 `/notes/*` 与 `/graphs/*` 会被识别为可预览引用。
- 其他链接不会被误拦截。
- diary 详情页使用了专用 diary 引用渲染路径，而不是继续直接依赖通用详情页布局。
- 浮层支持关闭按钮、遮罩关闭和 `Esc` 关闭。
- 引用目标不存在时会显示明确错误。
- `npm test` 与 Windows 环境 `npm run build` 通过。

## 实施边界结论

这次改造是一次受控的 diary 专项优化，而不是全站内容预览系统重构。

最终产物应满足两点：

1. `diary` 看起来像日志，而不是普通文章卡片页。
2. 作者可以在 diary 中自然提到某篇学习笔记或某个知识图谱，读者点击后能在当前页面内快速阅览，不打断日志阅读。
